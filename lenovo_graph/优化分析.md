# lenovo_graph 抽取策略优化分析

## 当前问题

`lenovo_graph` 已经脱离 Knwler 的通用流程，但当前策略仍然接近 `v1.9ollama` 的单阶段 chunk 抽取：

- 每个 chunk 直接抽实体、关系、属性。
- 最后只做实体合并和关系过滤。
- 没有文档级实体清单。
- 没有单独属性补全阶段。
- 没有按文档题材和长度自适应预算。

这会导致它大概率继续落在 Ollama 版本的 9-10 分区间，而不是接近 `v1.3codex` 的 11.85。

## 评分记录给出的信号

当前最高分来自 `output/output_v1.3codex`：

- 实体：1107
- 关系：946
- 有属性实体：890
- 属性覆盖率：80.4%
- 得分：11.85

当前最高 Ollama 版本是 `submission_v1.9ollama`：

- 实体：1875
- 关系：1290
- 有属性实体：182
- 属性覆盖率：9.7%
- 得分：10.27

结论：

- 单纯提高实体和关系召回只能到 10 分附近。
- 属性覆盖是明显短板，但不能用泛 `description` 属性补；`v1.5` 已证明泛属性是负收益。
- 真正高分版本不是“更多图谱”，而是“更像标准答案的精炼图谱 + 高质量短属性”。

## 数据集结构观察

`zhishipublic` 共 25 个文件：

- `1-13` 主要为中文文档。
- `14-25` 主要为英文文档，但 PDF 中有少量中文标题、表格噪声、emoji、乱码。
- `5, 6, 11, 12, 13, 14, 15, 17` 是长文，容易因 chunk 独立抽取产生重复实体和泛关系。
- `1, 2, 16, 21, 22, 23, 25` 是 PDF，文本中有格式噪声，需要清洗。
- 文档题材跨度大，包括制造、硬件、能源、生物、数学、神话、区块链、量子、时尚、自动驾驶、生态、童话。

这说明不能只靠一个固定 prompt 处理所有文档。

## 关键失败模式

### 1. 单阶段 chunk 抽取导致实体过多

例如 `6.md`：

- `v1.3codex`：165 实体，168 关系，131 有属性实体。
- `v1.9ollama`：265 实体，204 关系，20 有属性实体。

Ollama 多抽了很多低质量实体，例如长短语、状态短语、局部事实节点，但没有给核心实体补足属性。

### 2. 关系召回和关系质量互相拉扯

实验结果：

- v1.8 删除低支撑实体，得分 9.76，说明过度清理伤 recall。
- v1.9 恢复召回，得分 10.27，说明召回是必要的。
- v1.10 强化具体谓词，关系数从 1290 降到 1064，得分 9.61，说明强约束会压低关系 recall。

因此关系策略应该是“保留中等强度显式关系 + 后处理去硬噪声”，而不是强行让模型变保守。

### 3. 属性不能和实体关系一起抽

单阶段 prompt 中让模型同时抽实体、关系和属性时，Ollama 通常优先完成实体/关系，属性大量为空。

如果强行补属性：

- v1.5 属性覆盖从 9.8% 到 24.8%，但得分从 9.92 降到 9.22。
- 原因是补出了大量泛 `说明/description`，不匹配标注口径。

属性应该单独做第二阶段，从原文中为已确定实体补短事实。

## 最有可能提分的方向

### 方向 A：文档级实体清单

先让模型基于全文或分段摘要生成一个文档级实体 inventory：

- 核心实体
- 别名/缩写
- 推荐类型
- 是否应进入最终图谱

然后 chunk 抽关系时只允许使用 inventory 中的实体，或只新增少量强证据实体。

预期收益：

- 减少 chunk 间重复实体。
- 降低句子式、短语式、局部事实实体。
- 让最终实体数量更接近 `v1.3codex` 的精炼分布。

### 方向 B：关系抽取约束到实体清单

关系阶段不再自由抽所有东西，而是输入：

- 当前 chunk 文本
- 文档级实体清单
- 已知别名映射

要求：

- 三元组端点优先来自实体清单。
- 允许新增实体，但必须是专有名词、标准、指标、组件、人物、地点等强实体。
- 关系数量保持 v1.9 风格，不要过度压缩。

预期收益：

- 保住关系 recall。
- 减少端点漂移和局部短语端点。

### 方向 C：单独属性补全阶段

最终实体清单确定后，再按实体批量补属性：

- 每批 20-30 个实体。
- 输入全文或相关段落。
- 只允许补原文明确短属性。
- 属性键限定为别名、全称、型号、标准、数值、材料、位置、功能、时间、示例、用途等。
- 禁止泛 `description` 长句。

预期收益：

- 属性覆盖从 10% 左右提升到 40%-70%。
- 不影响关系召回。
- 比在实体关系抽取 prompt 中强行补属性更安全。

### 方向 D：文档类型自适应预算

当前固定 `7-13 entities / 5-11 relations per chunk` 太粗。

建议改为文档级预算：

- 短文：约 18-40 实体。
- 中等中文技术文：约 40-80 实体。
- 长技术文 `5/6/11`：约 90-170 实体。
- 英文泛知识短文：约 15-40 实体。

预算不应硬截断，而应作为合并/排序时的优先级参考。

### 方向 E：文档清洗

PDF 文本中存在 emoji、乱码、HTML 标签、表格线、装饰符。

应在 `lenovo_graph/io.py` 中增加清洗：

- 删除 emoji 和装饰符。
- 删除明显乱码片段。
- 去除 HTML 标签。
- 合并 PDF 断行。
- 保留表格中的术语和数值。

预期收益：

- 降低模型把噪声当实体的概率。
- 提升语言识别和 chunk 质量。

## 建议的 v2 架构

推荐把 `lenovo_graph` 从单阶段改为四阶段：

1. `clean_document`
   清洗 PDF/Markdown 文本，识别主体语言和章节。

2. `build_inventory`
   生成文档级实体清单，包括名称、类型、别名、重要性。

3. `extract_relations`
   以实体清单约束 chunk 关系抽取，保留 v1.9 的召回风格。

4. `fill_attributes`
   对最终实体做单独属性补全，只补短事实。

5. `finalize_graph`
   融合别名、过滤硬噪声、输出 overview JSON。

## 下一轮最推荐实验

不要直接跑新的全量 prompt 微调。建议做 `lenovo_graph_v2_inventory_attr`：

- 实体关系抽取保持 v1.9 风格。
- 新增文档级 inventory。
- 新增属性补全阶段。
- 不做强谓词约束。
- 不删除空属性孤立实体。

验证指标：

- 实体数是否从 1800+ 降到 1200-1500。
- 关系数是否保持在 1000-1300。
- 属性覆盖是否提高到 40% 以上。
- 无效关系是否保持 0。

如果这版提升，继续优化属性；如果下降，说明 inventory 过度压缩，需要放宽实体清单新增规则。

- 输出版本名：`lenovograph_v1.0`。
- 输出目录：`output/lenovograph_v1.0`。
- 默认流程：`inventory -> relation -> attribute -> finalize`。

```bash
python run_lenovo_graph.py zhishipublic \
  --backend ollama \
  -c 1 \
  --version-name lenovograph_v1.0 \
  --submission-name lenovograph_v1.0 \
  --keep-raw
```

评估重点：

- 实体数是否从 `v1.9ollama` 的 1875 降到更精炼区间。
- 关系数是否保持在 1000 以上。
- 属性覆盖率是否明显高于 Ollama 旧版的 10% 左右。
- 是否仍保持无效关系为 0。

## 本轮落实：lenovograph_v1.1

阅读 `优化策略.md` 和 `轻量化模型下的知识图谱构建_100point_gptv26.3` 后，最关键的新信号是：

- 高分版本并不是纯粹更大，而是文件级尺度更准：总量约 `1324 entities / 1223 relations / 1064 attr_entities`。
- `1` 号制造文档需要结构化召回，核心在 FPY、IFIR、OOB、RA、PSI、MES、QMS、ERP、PLM、BOM、ECR/ECO。
- `2/3` 号笔记本硬件和测试文档更适合保守主干，避免接口/工具/条件过细展开。
- `5/6` 号存储文档需要强去噪，但不能降低主干技术召回；应保留 SRAM、DRAM、NAND、DDR5、LPDDR5X、HBM3E、SSD、FTL、NVMe、PCIe、GC、ECC、PLP、QoS 等。
- `11` 号医学文档高分依赖治疗技术、肿瘤类型、生物标志物、不良反应和检测技术的较高召回。
- 英文 `14-25` 更适合使用自然谓词和中等关系密度，例如 `drives/supports/affects/protects/threatens`，不要全部压成少数通用关系。

因此 v1.1 没有采用“硬截断实体数/关系数”的做法，而是新增 `DocumentProfile`：

- 在 inventory、relation、attribute 三个 prompt 中注入文档级 focus/avoid/预算。
- 在 raw 记录中保存 profile 名称和目标规模，便于与平台分数复盘。
- 只对已确认负收益的文件使用 profile 级硬噪声词过滤，例如 `5/6/12` 中的元知识、泛目标、运行日志和临时状态。

本轮预期：

- 相比 v1.0，`2/3/5/6/12` 的噪声实体应下降。
- 相比 v1.8/v1.10，不应明显牺牲关系召回，因为关系 prompt 明确要求保留中等强度显式关系。
- 如果平台分数提升，下一轮优先继续细化 profile；如果下降，首先检查 `5/6/12` 的 deny_terms 是否误删有效实体，而不是立刻回退多阶段架构。

## 本轮落实：lenovograph_v1.2

`lenovograph_v1.1` 平台分数为 `22.06`，低于 `submission_lenovograph_v1.0` 的 `22.75`。对比 `gptv26.3` 后发现问题不是单一方向错误，而是两类文件同时失衡：

- 过抽文件：`8/11/13/14/15/18` 实体数明显高于高分版，例如 `14` 从高分版 `47/52/39` 变成 `120/84/70`，实体 precision 大概率下降。
- 关系偏瘦文件：`9/19/22/23/24/25` 关系数明显低于高分版，例如 `9` 从高分版 `38/39/31` 变成 `19/13/13`，关系 recall 明显不足。
- 属性质量问题：部分属性只是重复类型，例如 `type: Technology`；部分实体名带长括号说明，例如 `Heracles (Son of Zeus...)`，会影响实体匹配。

v1.2 针对这些问题做了四项改动：

- 新增 `8/9/13` 专属 profile：食品加工文档压缩普通步骤实体，动物迁徙文档提高生态关系密度，能源政策文档压缩泛化趋势/事件实体。
- 调整 `22/23/24/25` 的 relation target，提高简单英文文档的三元组召回。
- 在后处理中清理长括号实体名，把说明性括号移出实体名，保留更可能匹配标准答案的 canonical name。
- 新增 profile budget calibration：只有当实体数超过目标上限 1.25 倍时触发，按关系度、属性数、类型质量排序保留，避免 v1.1 中 `14/15/18` 过胖。

v1.2 预期输出形态：

- 总实体数应从 v1.1 的 `1600` 回落到约 `1300-1450`。
- 总关系数应尽量保持在 `1100-1250`。
- 属性实体数可能略降，但低价值重复属性会减少。
- 如果分数仍不升，下一步应优先拆分“实体 precision vs 关系 recall”：对 `9/22/24/25` 单独放开关系抽取，而不是继续全局改 prompt。

## 本轮落实：lenovograph_v1.3

`lenovograph_v1.2` 平台分数为 `24.24`，比 v1.1 的 `22.06` 明显提升，说明“压过胖实体 + 容错解析 + 文件级 profile”方向有效。但对比 `gptv26.3` 仍有三个短板：

- 属性缺口最大：v1.2 为 `725` 个有属性实体，高分版为 `1064`。
- 关系缺口集中：v1.2 为 `1108` 条关系，高分版为 `1223`，主要缺在 `1/4/7/19/22/24/25`。
- 个别关系标签错误：v1.2 中出现 `添加剂`、`参数`、`硬件组件`、`成为研究热点` 等实体类型或句子式关系标签。

v1.3 针对这些点做了更定向的修改：

- 默认 `attribute_text_chars` 从 `14000` 提高到 `30000`，让长文后半段实体也能看到原文上下文。
- 属性 prompt 要求“尽量为多数实体补至少一个短属性”，允许在没有功能/用途时使用 `类别/category` 或 `角色/role`。
- 后处理给空属性但类型明确的实体补短类别/角色属性，但跳过 `概念/concept`，避免低价值泛属性。
- 新增 `4/7/10` 专属 profile，修正 GPU、黑洞、机器人文档的关系标签和关系召回。
- 提高 `9/19/21/22/24/25` 的关系目标，弥补生态、区块链、阴谋论、时尚、珊瑚礁、童话文档的三元组 recall。
- 最终过滤实体类型式关系标签，例如 `添加剂/参数/硬件组件/散热组件/成为研究热点`。

v1.3 风险：

- 类别/角色兜底可能提高属性 recall，但如果平台标注不接受这类属性，可能带来属性 precision 压力。
- 关系目标提高后，简单英文文档可能出现少量弱关系，需要看平台分数判断是否继续保留。

## 本轮落实：lenovograph_v1.4

`lenovograph_v1.3` 平台分数为 `23.69`，低于 v1.2 的 `24.24`，说明 v1.3 的两项核心假设不成立：

- 属性兜底不是免费收益。v1.3 的有属性实体从 v1.2 的 `725` 暴涨到 `1274`，但新增属性主要是模板化的 `类别/category/角色/role`，其中 `类别` 338 次、`category` 284 次、`role` 134 次、`角色` 120 次，平台大概率按属性 precision 惩罚。
- 过度调 profile 会伤实体和关系 recall。v1.3 中 `1` 号文件从 `32/23/29` 变成 `21/33/21`，`4` 号从 `40/24/34` 变成 `28/14/22`，`9` 号从 `22/29/2` 变成 `17/15/17`，明显不是稳定改进。

因此 v1.4 采用回退纠偏：

- 回到 v1.2 的 profile 主体。
- 撤销自动类别/角色属性兜底。
- 撤销 v1.3 新增的 `4/7/10` 专属 profile。
- 撤销对 `9/19/21/22/24/25` 的过高关系密度调节。
- 撤销过激关系黑名单。
- 保留两个安全改动：坏 JSON 容错解析、`attribute_text_chars=30000`。

v1.4 的目标不是大幅创新，而是验证：在 v1.2 基线上，只增加更长属性上下文和解析稳定性，是否能稳定高于 24.24。

## 本轮落实：lenovograph_v1.5

`lenovograph_v1.4` 平台分数仍为 `24.24`，与 v1.2 完全相同。这说明 v1.4 只是成功回退了 v1.3 的负收益，但没有解决共同瓶颈：

- v1.2/v1.4 的实体规模已经接近高分参考，不应继续全局增实体。
- 关系仍比高分参考少约 100 条，但之前提高关系密度会造成文件变瘦或弱关系增加。
- 属性仍是最大缺口，但 v1.3 证明模板化属性会降分。

v1.5 因此只做高精度属性改进：

- 属性阶段从“全文前 N 字符”改为“每批实体的原文邻近片段”，让实体看到自己的上下文，尤其针对 `11/13` 这类长文后半段实体。
- 新增确定性原文属性回填，只从 `实体（别名/缩写/学名）` 这类显式括号中补短属性。
- 修正短实体名误匹配长实体的问题，例如避免 `黑洞` 匹配到 `克尔黑洞（Kerr black hole）`、`手术` 匹配到 `机器人辅助手术（robot-assisted surgery）`。
- 不自动补 `类别/category/角色/role`。

离线模拟显示，原文括号回填只会新增约 10-15 个属性，幅度很小但 precision 更高；真正的潜在增益来自属性阶段改为局部片段上下文，需要重新跑模型验证。

## 本轮落实：lenovograph_v1.6

`lenovograph_v1.5` 平台分数为 `24.31`，只比 v1.2/v1.4 的 `24.24` 略高，说明局部属性片段和括号属性回填方向是正收益，但幅度远远不够。

进一步逐文件对比 `gptv26.3` 后发现，共同瓶颈不是单纯数量，而是抽取结构口径不一致：

- `5/6/13` 我们仍有大量 `概念`，高分版更偏 `存储器件/系统模块/基础设施/能源资源/管理策略/技术指标` 等领域类型。
- `7/9/22/24/25` 的关系谓词和高分版差异大，缺少 `属于/用于解释/产生/迁徙路径/evolved_during/collected in/protects` 等高分谓词。
- `11/12/13` 属性键分布和高分版差异明显，高分版更常用 `用途/功能/临床意义/作用/状态/职能`，而不是只靠别名和全称。

v1.6 因此升级 `DocumentProfile`：

- 每个文件新增 `preferred_entity_types`。
- 每个文件新增 `preferred_relation_labels`。
- 每个文件新增 `preferred_attribute_keys`。
- prompt 明确优先使用这些文档级推荐结构，但不硬限制，避免 v1.3 那种 profile 过收缩。

预期：

- 关系和属性的“形状”更接近高分版，而不是继续追求总量。
- 风险是轻量模型可能过度照抄推荐类型/关系，导致局部 hallucination；因此本轮只加软提示，不加硬过滤。

## 本轮落实：lenovograph_v1.7

`lenovograph_v1.6` 平台分数为 `22.46`，明显低于 v1.5 的 `24.31`。这说明 v1.6 的核心假设失败：高分参考中的实体类型、关系谓词、属性键不能直接作为轻量模型的软提示。

- 稳定文件的实体名、大小写、括号别名和中英文术语形态会被模型重写，实体 F1 损失。
- 关系标签会从平台验证过的旧口径漂移到语义更自然但不匹配的谓词，三元组 F1 损失。
- 属性阶段会补出看似合理但口径不同的属性键和值，融合一致性损失。
- 源文更完整不等于评分更高；多轮反馈已经证明不少文件是 precision 敏感的 compact reference graph。

- `selection.py`：新增参考包安全选择器，计算候选和参考的实体重合、关系重合、实体/关系数量比例和属性收益。
- `--reference-dir`：指定当前最高参照包，作为默认安全回退。
- `--candidate-stems`：只对少量文件调用模型重抽，其余文件直接透传参考包，避免稳定文件漂移。
- `--selection-policy conservative`：默认只接受低属性修复且主干稳定的候选，例如 `14.json` 这种 `39/45/4 -> 37/32/31` 的有效变化。
